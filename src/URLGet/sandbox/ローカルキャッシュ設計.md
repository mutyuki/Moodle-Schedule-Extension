# 授業コード・Salesforce ID ローカルキャッシュ設計

## 1. 目的

Chrome拡張機能が同じ授業情報を表示するたびにAPIやデータベースへアクセスすることを防ぐ。

一度取得した授業コードとSalesforce IDの対応情報を、Chrome拡張機能のローカル領域へ保存する。次回以降はローカルのデータを優先して使用し、データが存在しない場合や有効期限が切れている場合のみAPIへアクセスする。

利用者はIT初心者を想定し、キャッシュに関する設定や操作を一切求めない。拡張機能のインストール後は、Moodleを通常どおり利用するだけで自動的にキャッシュを保存・更新する。

## 2. 利用者に設定を求めない設計方針

利用者が行う操作は、Chrome拡張機能のインストールのみとする。インストール後の初期設定、保存先の選択、キャッシュ更新、キャッシュ削除などは不要とする。

拡張機能が自動的に行う処理:

- Moodleの時間割を開いたときに授業情報を読み取る
- ローカルキャッシュの有無と有効期限を確認する
- 必要な場合のみAPIからシラバス情報を取得する
- APIの取得結果を自動的に保存する
- 有効期限が切れたデータを自動的に更新する
- 古い年度や不要になったデータを自動的に削除する
- 通信や保存に失敗した場合、Moodleの通常利用を妨げずに処理を終了する

利用者向けの設定画面やキャッシュ管理画面は、原則として作成しない。

```text
拡張機能をインストール
  ↓
利用者がMoodleを通常どおり開く
  ↓
拡張機能が自動的にキャッシュ確認・API取得・保存を行う
  ↓
シラバスへのリンクを表示する
```

## 3. 使用する保存領域

ブラウザ標準の`localStorage`ではなく、Chrome拡張機能向けの`chrome.storage.local`を使用する。

`chrome.storage.local`を使用する理由:

- Content ScriptとService Workerの両方から利用できる
- 非同期APIのため、保存処理で画面をブロックしにくい
- Moodleページ側のローカルストレージと分離される
- ブラウザのキャッシュや閲覧履歴を削除しても基本的に維持される
- 通常10MBまで保存でき、この用途では十分な容量がある

`localStorage`はService Workerから利用できず、Content Scriptから使用するとMoodleページ側の保存領域を使用するため、この用途には適さない。

## 4. Manifestの設定

`chrome.storage.local`を使用するには、`manifest.json`へ`storage`権限を追加する。

この権限は拡張機能のインストール時に有効になるため、利用者がChromeの設定画面で保存先などを設定する必要はない。

```json
{
  "manifest_version": 3,
  "permissions": ["storage"]
}
```

## 5. キャッシュキー

Salesforce IDやシラバス情報は年度ごとに変わる可能性があるため、授業コードだけではなく年度もキャッシュキーに含める。

基本形式:

```text
syllabus:{年度}:{授業コード}
```

例:

```text
syllabus:2026:COURSE001
```

同じ年度・授業コードに複数のシラバスが存在する場合は、学期や担当教員名もキーに含める。

```text
syllabus:{年度}:{授業コード}:{学期}:{担当教員名}
```

キャッシュキーに使用する値は、APIの検索条件と一致させる。

## 6. 保存するデータ

```js
{
  "syllabus:2026:COURSE001": {
    courseCode: "COURSE001",
    salesforceId: "0123456789ABCDEFGH",
    syllabusUrl: "https://example.com/syllabus/0123456789ABCDEFGH",
    expiresAt: 1782860400000
  }
}
```

保存項目:

| 項目 | 内容 |
|---|---|
| `courseCode` | 授業コード |
| `salesforceId` | Salesforceが発行する18桁のID |
| `syllabusUrl` | シラバス詳細ページのURL |
| `expiresAt` | キャッシュの有効期限を表すUnix時刻（ミリ秒） |

Salesforce IDは大文字と小文字を区別するため、保存時に変換しない。

## 7. キャッシュの保存・読み込み処理

```js
const CACHE_TTL_MS = 7 * 24 * 60 * 60 * 1000;

function createCacheKey(course) {
  return [
    "syllabus",
    course.academicYear,
    course.courseCode,
    course.semester ?? "",
    course.instructorName ?? ""
  ].join(":");
}

async function getCachedSyllabus(course) {
  const key = createCacheKey(course);
  const result = await chrome.storage.local.get(key);
  const cached = result[key];

  if (!cached) {
    return null;
  }

  if (cached.expiresAt <= Date.now()) {
    await chrome.storage.local.remove(key);
    return null;
  }

  return cached;
}

async function saveCachedSyllabus(course, syllabus) {
  const key = createCacheKey(course);

  await chrome.storage.local.set({
    [key]: {
      courseCode: course.courseCode,
      salesforceId: syllabus.salesforceId,
      syllabusUrl: syllabus.syllabusUrl,
      expiresAt: Date.now() + CACHE_TTL_MS
    }
  });
}
```

## 8. API取得処理との組み合わせ

データ取得時は、最初にローカルキャッシュを確認する。

```js
async function resolveSyllabus(course) {
  const cached = await getCachedSyllabus(course);

  if (cached) {
    return cached;
  }

  const syllabus = await fetchSyllabusFromApi(course);
  await saveCachedSyllabus(course, syllabus);

  return syllabus;
}
```

処理の流れ:

```text
授業情報を取得
  ↓
chrome.storage.localを検索
  ↓
有効なキャッシュが存在する
  ├─ はい: キャッシュを使用する
  └─ いいえ: APIへアクセスする
                 ↓
              取得結果を保存する
                 ↓
              取得結果を使用する
```

## 9. `not_found`のキャッシュ

APIから`not_found`が返った場合も短時間キャッシュすることで、存在しない授業に対する繰り返しアクセスを防止できる。

ただし、データベースへ新しいシラバス情報が追加される可能性があるため、正常取得時より短い有効期限を設定する。

推奨する有効期限:

| 結果 | 有効期限の例 |
|---|---|
| `found` | 7日 |
| `not_found` | 1時間 |
| `ambiguous` | 原則キャッシュしない、または短時間 |
| APIエラー | キャッシュしない |

## 10. 同時リクエストへの対策

同じページ内で同一授業の取得処理が同時に実行されると、キャッシュ保存前に複数回APIへアクセスする可能性がある。

処理中のPromiseを授業ごとに保持し、同じ授業のAPIリクエストを共有する。

```js
const pendingRequests = new Map();

async function resolveSyllabusOnce(course) {
  const key = createCacheKey(course);

  if (pendingRequests.has(key)) {
    return pendingRequests.get(key);
  }

  const request = resolveSyllabus(course).finally(() => {
    pendingRequests.delete(key);
  });

  pendingRequests.set(key, request);
  return request;
}
```

このMapはメモリ上にのみ存在するため、Service Workerが停止した場合は消える。永続キャッシュには引き続き`chrome.storage.local`を使用する。

## 11. キャッシュ削除

利用者に削除操作を求めず、拡張機能が自動的にキャッシュを削除する。

- 有効期限が切れた場合
- 保存形式を変更した場合
- 誤ったシラバス情報を修正した場合
- 保存件数が上限を超えた場合
- 対象年度から一定期間が経過した場合

保存形式を変更するときはキャッシュデータにバージョン番号を持たせ、古い形式を自動削除する方法を推奨する。

```js
const CACHE_VERSION = 1;
```

シラバスキャッシュだけを削除する例:

```js
async function clearSyllabusCache() {
  const items = await chrome.storage.local.get(null);
  const keys = Object.keys(items).filter((key) => key.startsWith("syllabus:"));

  await chrome.storage.local.remove(keys);
}
```

`chrome.storage.local.clear()`は拡張機能の他の設定も削除するため、シラバスキャッシュだけを削除したい場合は使用しない。

## 12. エラー発生時の表示方針

通信やキャッシュ保存に失敗した場合も、利用者へ技術的なエラーや設定作業を求めない。

- Moodle本来の表示と操作を妨げない
- キャッシュ保存に失敗しても、取得できたシラバスリンクは可能な範囲で表示する
- APIへ接続できない場合は、シラバスリンクを非表示にするか「現在取得できません」と簡潔に表示する
- 自動再試行は回数と間隔を制限し、APIへ過剰にアクセスしない
- 詳細なエラーは開発者向けログへ記録し、通常の利用者には表示しない

## 13. 注意点

- キャッシュキーには、シラバスを一意に特定するために必要な検索条件をすべて含める
- Salesforce IDの大文字・小文字を変更しない
- キャッシュはAPIやデータベースの代わりではなく、負荷を減らすための一時データとして扱う
- APIエラーを正常な検索結果として保存しない
- 保存・読み込み処理の失敗時もMoodleページの表示を壊さない
- 大量の授業情報を一括取得する場合は、`chrome.storage.local.get()`と`set()`の一括操作を検討する
- 利用者にキャッシュの存在を意識させない
- 利用者に年度、Salesforce ID、有効期限などを入力させない
- 通常利用に不要な設定画面や確認ダイアログを表示しない

## 14. テスト項目

- 拡張機能のインストール後、利用者による初期設定なしで動作する
- 初回表示時にAPIへアクセスし、取得結果が保存される
- 2回目の表示時はAPIへアクセスせず、保存済みデータが使用される
- 有効期限切れの場合はAPIから再取得される
- 年度が異なる同じ授業コードを別データとして保存できる
- Salesforce IDの大文字・小文字が維持される
- `not_found`への繰り返しアクセスが抑制される
- 同一授業の同時取得でAPIリクエストが重複しない
- 不要なキャッシュが自動削除され、その後APIから再取得される
- ストレージ操作やAPI通信に失敗してもMoodleを通常どおり利用できる
- エラー発生時に利用者へ技術的な設定作業を求めない
