# URLGet キャッシュ機能について

## 1. これは何？

`URLGet/sandbox/content.js` は、Moodleの時間割表から授業コードを取得し、Cloudflare Workers上のAPI (`https://withered-salad-b4aa.yudai-syllabus.workers.dev/syllabus`) に問い合わせて、シラバスのURLを生成する拡張機能です。

このAPIを毎回呼び出すと、利用者が増えたときにAPIへの負荷が大きくなってしまいます。そこで、一度取得したシラバスURLを**ブラウザの中に保存（キャッシュ）**しておき、同じ授業コードであれば次回からはAPIを呼ばずに保存されたデータを使うようにしました。

利用者が何か設定をする必要はありません。Moodleを通常どおり開くだけで、自動的にキャッシュの確認・保存が行われます。

## 2. 仕組み

### 使っている保存場所

`chrome.storage.local` というChrome拡張機能用の保存領域を使っています。
（`manifest.json` に `"storage"` という権限を追加することで使えるようになります）

### 処理の流れ

```text
授業コードを取得
  ↓
chrome.storage.local に同じ授業コードのデータがあるか確認
  ↓
┌─ ある（かつ有効期限内） → 保存されているURLをそのまま使う
│
└─ ない（または有効期限切れ） →
       APIへアクセスしてシラバスURLを取得
         ↓
       取得したURLを chrome.storage.local に保存
         ↓
       そのURLを使う
```

### 保存されるデータの形

授業コードごとに、以下のような形式で保存されます。

```js
{
  "syllabus:ABC123": {
    courseCode: "ABC123",
    syllabusUrl: "https://syllabus.ritsumei.ac.jp/syllabus/s/r-syllabus/0123456789ABCDEFGH",
    expiresAt: 1782860400000 // 有効期限（Unix時刻・ミリ秒）
  }
}
```

- キー名は `syllabus:` + 授業コード (例: `syllabus:ABC123`)
- `expiresAt` を過ぎると、そのキャッシュは無効になり、削除されてAPIから再取得されます

### 有効期限（TTL）

保存されたキャッシュは **7日間** 有効です。7日を過ぎると自動的に削除され、次回アクセス時にAPIから新しいデータを取得・保存し直します。

```js
const CACHE_TTL_MS = 7 * 24 * 60 * 60 * 1000; // 7日間
```

### 失敗してもMoodleは止まらない

キャッシュの読み込み・保存に失敗しても、エラーをコンソールに出すだけで処理を止めません。失敗した場合はAPIから直接取得するので、Moodleの通常利用は妨げられません。

また、APIから正しい結果（`salesforce_id`）が得られなかった場合は、キャッシュに保存しません。誤った情報がキャッシュされ続けることを防ぐためです。

## 3. 動作確認方法

1. Chromeで `chrome://extensions` を開き、デベロッパーモードを有効にする
2. 「パッケージ化されていない拡張機能を読み込む」で `URLGet/sandbox` フォルダを選択する
3. Moodleの時間割ページを開く
4. デベロッパーツール（F12）を開き、以下を確認する
   - **Networkタブ**: 初回表示時にAPI (`.../syllabus?code=...`) へのリクエストが発生していること
   - **Application（または Storage）タブ → Extension Storage → Local Storage**: `syllabus:授業コード` というキーでデータが保存されていること
   - **Consoleタブ**: 初回表示時は授業コードごとに `[API] 授業コード: URL` というログが出力されていること
5. ページを再読み込みする
   - 2回目以降は、保存済みの授業コードについてAPIへのリクエストが発生しないこと
   - Consoleタブで、保存済みの授業コードについて `[キャッシュ] 授業コード: URL` というログが出力されていること
   - コンソールの「生成したシラバスURL一覧」が、前回と同じ結果になること

## 4. キャッシュのリセット方法

通常は何もしなくても、7日間（`CACHE_TTL_MS`）が過ぎると古いキャッシュは自動的に削除され、APIから再取得されます。ただし、動作確認や不具合調査のために、手動でキャッシュをリセットしたい場合は以下の方法を使います。

### 方法1: 特定の授業コードだけ削除する（DevTools操作）

1. Moodleページでデベロッパーツール（F12）を開く
2. **Application（または Storage）タブ → Extension Storage → Local Storage** を開く
3. 削除したい `syllabus:授業コード` の行を選び、削除する
4. ページを再読み込みすると、その授業コードだけAPIから再取得される

### 方法2: シラバスキャッシュを全部まとめて削除する（Consoleで実行）

DevToolsの **Console タブ** で、以下のコードを貼り付けて実行すると、`syllabus:` から始まるキーだけをまとめて削除できます。

```js
// "syllabus:" で始まるキャッシュだけをすべて削除する
chrome.storage.local.get(null, (items) => {
  const keys = Object.keys(items).filter((key) => key.startsWith('syllabus:'));
  chrome.storage.local.remove(keys, () => {
    console.log('削除したキー:', keys);
  });
});
```

実行後にページを再読み込みすると、すべての授業コードがAPIから再取得され、キャッシュが作り直されます。

> `chrome.storage.local.clear()` は使わないでください。シラバスのキャッシュ以外に、拡張機能が使っている他の設定もすべて削除されてしまいます。

### 方法3: 拡張機能を削除・再読み込みする

`chrome://extensions` で拡張機能を削除（または「削除」→ 再度「パッケージ化されていない拡張機能を読み込む」）すると、`chrome.storage.local` の内容もすべて初期化されます。動作確認用に拡張機能ごとリセットしたい場合に使えます。

### 方法4: 何もしない（自動リセットを待つ）

保存から7日（`CACHE_TTL_MS`）が経過すると、そのキャッシュは次回アクセス時に自動的に無効と判定され、削除・再取得されます。利用者が操作する必要はありません。

## 5. `ローカルキャッシュ設計.md` との対応関係

このリポジトリには、より詳細なキャッシュ設計をまとめた `ローカルキャッシュ設計.md` があります。今回の実装は、その設計のうち以下の部分を反映しています。

- `chrome.storage.local` を使ったキャッシュ保存・読み込み（設計書セクション3, 7, 8）
- 有効期限（TTL）に基づくキャッシュの自動失効・削除（設計書セクション7, 11）
- API取得結果のうち、正しいデータのみキャッシュする（設計書セクション9, 13）
- ストレージ操作・API通信が失敗してもMoodleの利用を妨げない（設計書セクション12）

一方で、以下は今回は実装していません（現状の `content.js` では授業コード以外の情報を取得していないため）。

- キャッシュキーへの年度・学期・担当教員名の追加（設計書セクション5）
  - 現在は `syllabus:{授業コード}` のみでキャッシュしています
  - 将来、年度をまたいで同じ授業コードが再利用される場合に古いURLが一時的に残る可能性がありますが、7日間のTTLにより一定期間後には自動的に再取得されます
- `not_found` / `ambiguous` といったAPIの応答パターンごとの個別キャッシュ（設計書セクション9）
- 同時リクエストの重複防止（設計書セクション10）
